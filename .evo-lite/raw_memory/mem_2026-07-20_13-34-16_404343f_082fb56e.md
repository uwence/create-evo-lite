## 实现细节 (Implementation)
[bug-fix]
Follow-up to da53d3d. CodePLC re-dogfooded the nurtured fix and found a second, adjacent gap: templates/cli/test.js's entry point calls runIntegrationTests() unconditionally after the governance scope returns. The integration suite loads mother-only modules (e.g. templates/cli/planning/progress.js via TEMPLATE_CLI_DIR) that do not exist in a child hive checkout, so a child running the default 'all' scope (governance test scope arg omitted, as CodePLC's own test runner does) passed governance then crashed loading the integration suite and exited 1 -- CodePLC correctly declined to commit the previous nurture receipt over this. Root cause: an old copy of this file (visible in git history predating the last nurture) DID import IS_CHILD_RUNTIME from ./test/harness and returned early before runIntegrationTests(); that check was dropped in a later mother-side edit, silently reintroducing the exact failure mode T27/T-spec-cli guarding was meant to prevent. Fixed by re-importing IS_CHILD_RUNTIME and returning before runIntegrationTests() when true, with the skip message matching the existing governance-suite skip style. Added T-entry-child-guard: a static source-text regression test (reads via CLI_DIR so it runs from either the mother's templates/cli or a nurtured child's .evo-lite/cli) asserting the guard exists and sits before the runIntegrationTests() call, so this can't silently regress a second time. Verified end-to-end in a from-scratch bare-child simulation (fresh temp dir with only the synced .evo-lite/cli tree, no templates/ sibling, node_modules symlinked): node test.js all (no scope arg, matching how CodePLC invokes it) now exits 0, printing both the governance child-mode skip and the new integration-suite child-mode skip. Full mother-mode 'all' scope also green (no regression).
