# Release 2.2.0 Hardening — Trust Boundary Completion, Hive Truth Source, Release Closure

## Context

Two independent assessments of `main` @ `44b3609` (2026-07-06) converged:

1. **Dogfood retrospective (internal).** Detection is strong (drift, verify
   gates, fail-closed evidence) but the remaining friction clusters in three
   places: closure is manual, failures can be silent to a non-expert operator,
   and the mother keeps growing governance features while the one real child
   (hungersnakegame4) already produced 7 unaddressed nurture pain points.
2. **External code review.** Found that the P1-8 command-verifier trust
   boundary, while correctly default-denying spec-authored `cmd` strings, does
   NOT cover two adjacent input paths, plus hive/nurture correctness gaps and
   an incomplete release state.

Every review claim below was re-verified against the working tree before this
spec was written:

- **CONFIRMED (P0)** — evidence `commitSha` reaches a shell via string
  interpolation: [`engine.js:76`](../../../templates/cli/verification/engine.js)
  does `` exec(`git diff ${sha}..HEAD --name-only`) `` where `sha` comes from
  an evidence record, and [`evidence-store.js:26`](../../../templates/cli/verification/evidence-store.js)
  `readEvidence()` is a bare `JSON.parse` with no record validation. A
  tampered evidence file (`"commitSha": "HEAD; <cmd> #"`) executes arbitrary
  shell during `mem verify-contract status`. Same trust class P1-8 just
  closed, different door.
- **CONFIRMED (P1)** — hive reads the wrong version truth source:
  [`hive/status.js:41`](../../../.evo-lite/cli/hive/status.js) reads
  `.evo-lite/package.json` (runtime manifest, pinned `1.0.0` by design) while
  the product version of a scaffolded child lives in
  `.evo-lite/evo-lite-version.json` (written by [`index.js:50`](../../../index.js),
  preferred by [`runtime.js:26`](../../../templates/cli/runtime.js)). A fresh
  child therefore always reports `behind (1.0.0 → <mother>)`, and
  `nurtureChild()` "fixes" it by mutating `package.json.version`
  ([`hive/nurture.js:150`](../../../.evo-lite/cli/hive/nurture.js)) — breaking
  the deliberate manifest/product decoupling and desyncing the lockfile.
- **CONFIRMED (P1)** — nurture's apply phase (overwrite genes → mirror lock →
  receipt → child pkg → mother registry) has no journal/rollback; any mid-write
  failure leaves a half-nurtured child. `close-apply.js` already implements
  the correct snapshot/journal/rollback pattern; nurture does not reuse it.
- **CONFIRMED (P2)** — [`contract-schema.json`](../../../templates/cli/verification/contract-schema.json)
  declares `cwd`/`scope` as optional params of the `command` verifier, but
  [`run-verifiers.js:78`](../../../templates/cli/verification/run-verifiers.js)
  only reads `p.cmd` and always executes at `repoRoot`. Declared-but-dead
  contract surface.
- **CONFIRMED (P2)** — [`run-verifiers.js:15`](../../../templates/cli/verification/run-verifiers.js)
  `resolveWithin()` is string-prefix containment only; a symlink inside the
  repo pointing outside defeats it.
- **CONFIRMED** — `active_context.md` FOCUS says "7 commits unpushed" while
  `git rev-list @{u}..HEAD --count` = 0. The tool whose purpose is accurate
  project state carries stale machine-checkable state as free text.
- **CONFIRMED** — `CHANGELOG.md` `## Unreleased` is empty despite hive,
  nurture, command policy, and closure dogfood having landed since 2.1.0;
  README command reference lacks `mem verify-contract`, `mem close`,
  `mem hive`.

Review points **rejected or amended** (recorded so they don't resurface):

- The review proposed rule ID `R012 active-context-remote-drift`; R012 is
  already taken. This spec uses **R013**.
- The review proposed modeling four hive drift substates
  (`genes-up-to-date` / `product-version-behind` / `runtime-deps-drifted` /
  `runtime-not-ready`). Deferred: one child exists; substates before a second
  consumer is speculative surface. Tracked in Follow-up.
- The review's numeric scorecards are not actionable and are ignored.

## Goal

Close every confirmed trust/correctness gap above, ship an honest 2.2.0
release closure, and then **freeze mother-side governance expansion** — the
next specs after this one must come from child-driven pain points and the
human-facing perception layer, not from new engine capability.

## Non-Goals

- **Implementing `cwd` for command verifiers** — rejected. Executing at a
  spec-chosen directory is new attack surface with no current consumer.
  `cwd` is deleted from the schema instead. `scope` is kept but explicitly
  documented as an informational label (it appears inside `cmd` anyway);
  no execution semantics are attached to it.
- **Hive drift substates** — deferred until a second real child diverges
  (see Follow-up).
- **New verdict states** — INVALID evidence maps onto existing derived
  semantics (see Design); the stored-verdict invariant `{PASS, FAIL}` holds.
- **Sandboxing / containerization** — unchanged from spec:command-verifier-trust-boundary.
- **Any new mother-side governance feature not listed here.** This is a
  direction constraint, not just scope control: after 2.2.0, priority order is
  (1) hungersnakegame4 nurture pain points, (2) closure automation
  (`mem close` single-command evidence backfill), (3) perception layer
  (plain-language dashboard, no R-codes in operator-facing output),
  (4) a timed non-expert usability test ("agent went off-track — can the
  operator see it within 5 minutes?").

## Design

### 1. Evidence trust boundary (P0)

**No evidence-derived string ever reaches a shell.**

- `engine.js` gitDiff: replace string `exec` with
  `execFileSync('git', ['diff', '--name-only', `${sha}..HEAD`, '--'], { cwd: root, encoding: 'utf8' })`.
  Argv form kills injection even for malformed `sha`; the existing
  catch → `null` → STALE path is unchanged.
- `commitSha` format gate **before** any git call: must match
  `/^[0-9a-f]{40}([0-9a-f]{24})?$/` (40- or 64-hex OID). Non-conforming →
  treat the record's evidence as unusable → criterion derives **STALE**
  (never executed, never trusted), with a warning line naming the record.
- `readEvidence()` validates on read: top level must be the expected store
  shape; each record must have `criterionId` (string), `verdict` ∈
  `{PASS, FAIL}`, `commitSha` (OID format), and the digest fields the engine
  consumes. A malformed **file** (unparseable / wrong top-level shape) →
  throw, fail-closed, same policy as `loadPolicy`. An individually invalid
  **record** → excluded with a printed warning → the criterion is UNVERIFIED
  by absence or STALE per above. Silent exclusion is not allowed.
- Regression tests include a hostile fixture:
  `"commitSha": "HEAD; echo pwned #"` must produce STALE and (via injected
  exec spy) **zero** shell invocations.

### 2. Hive product-version truth source (P1)

- `childStatus()` reads `.evo-lite/evo-lite-version.json`; falls back to
  `.evo-lite/package.json` **only** when the version file is absent (legacy
  child), and marks that status line `(legacy version source)`.
- `nurtureChild()` writes the mother's product version to
  `.evo-lite/evo-lite-version.json` and **never mutates**
  `.evo-lite/package.json.version` (nurture.js:150 path removed).
- `registry.js` child detection unchanged (`package.json` existence is still
  a fine structural marker).
- Hive tests gain a fixture matching the real initializer product
  (`package.json = 1.0.0`, `evo-lite-version.json = <product>`): a fresh
  child must report **up-to-date**, and a post-nurture child must leave
  `package.json.version === '1.0.0'`.

### 3. Nurture atomicity (P1)

- Extract the snapshot/journal/rollback/lock mechanics of
  `close-apply.js` into a shared gene `templates/cli/transaction.js`
  (acquireLock / snapshotFiles / applyWrites / rollbackFiles / writeJournal /
  releaseLock).
- `close --apply` and `hive nurture` both consume it. Nurture's full apply
  set — gene files, mirror lock, receipt, `evo-lite-version.json`, mother
  registry — goes through one transaction: any failure restores every
  snapshotted file and reports the journal path.
- Until this lands, nurture output must not claim "all-or-nothing"
  (interim doc honesty is part of Task ordering in the plan).

### 4. Contract honesty (P2)

- Delete `cwd` from `contract-schema.json` `command.optionalParams`.
- Keep `scope`; document in the schema comment / spec-format doc that it is
  an informational label with no execution semantics.
- Sweep existing spec criteria: none use `cwd` (verify at plan time); if any
  do, migrate them in the same commit as the schema change.

### 5. Symlink-aware containment (P2)

- `resolveWithin()`: after prefix check, resolve reality — existing target →
  `fs.realpathSync(abs)` must still be contained; non-existing target →
  realpath of the nearest existing ancestor must be contained. Throw the
  same `path escapes project root` error otherwise.
- Test: repo-internal symlink to an outside directory; reading through it
  must throw, not read.

### 6. R013 active-context-remote-drift

Free text must not carry machine-checkable git state.

- Add structured fields to the `BEGIN_META` block of `active_context.md`
  (written only by the CLI, anchor-protected): `headSha`, `upstreamSha`,
  `ahead`, `behind`, `focusUpdatedAt`.
- New drift rule **R013** in `planning/gaps.js` (alongside R012, which already
  reads `active_context.md`), wired into `runPlanningDrift` so
  `inspector.js` and `mcp-server.js` surface it: fires when (a) META `headSha`
  is not HEAD or an ancestor of HEAD, or (b) META ahead/behind disagree with
  `git rev-list --left-right --count @{u}...HEAD`. Repos without an upstream
  skip (b).
- R013 checks **only** the structured META fields — it does not attempt to
  parse FOCUS prose. `context track` (in `memory.service.js`) refreshes the
  META git fields as part of its existing state snapshot; the
  `templates/active_context.md` scaffold gains the placeholder fields.

### 7. Release closure 2.2.0

- Version bump `2.1.0 → 2.2.0` (hive + nurture + command policy + closure
  dogfood are minor-grade, not patch).
- `CHANGELOG.md` Unreleased backfilled with everything since 2.1.0, then cut
  to `## 2.2.0`.
- README + README_EN: command reference gains `mem verify-contract …`,
  `mem close …`, `mem hive …`; hive and verification-contract get short
  operator-facing sections (plain language, no R-codes).
- Repo settings (human-executed, attested via manual verifier): main branch
  protection with `release-gate` as a **required** check; publish happens
  from tag/workflow, not local `npm publish`.
- Publish only after all other criteria of this spec are PASS on a clean
  HEAD.

## Testing

Governance suite (`test/governance.js`), new T-numbers:

1. Hostile `commitSha` fixture → STALE, exec spy never called (P0 regression).
2. `readEvidence` — malformed file throws; invalid record excluded with
   warning; valid store passes through byte-identical behavior.
3. OID gate — 40-hex passes, 64-hex passes, `HEAD`, empty, and
   metacharacter strings rejected.
4. Hive fixture with real initializer shape → up-to-date; legacy child
   (no version file) → fallback + `(legacy version source)` marker.
5. Nurture → `package.json.version` untouched; `evo-lite-version.json`
   updated; injected mid-apply failure → all snapshotted files restored.
6. `transaction.js` unit coverage reused by both consumers (close-apply
   regression suite must stay green unmodified).
7. Symlink escape fixture → throw.
8. Schema: `cwd` rejected as unknown param; `scope` still accepted.
9. R013: stale `headSha` fires; matching state silent; no-upstream repo
   skips the ahead/behind check.
