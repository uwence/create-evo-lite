# 🚀 响应 Evo 唤醒协议

收到即已唤醒，执行 Evo-Lite 专属操作。

步骤：
// turbo-all
1. 先**静默读取**本项目 `.evo-lite/active_context.md`，看当前进度。
   - 文件由 HTML 注释锚点分 4 个隔离块：`META`（项目元数据）、`FOCUS`（当前焦点）、`BACKLOG`（活跃任务 ≤5 条）、`TRAJECTORY`（最近轨迹 ≤10 条）。
   - **锚点守卫**：后续写入必须只落在对应锚点块内，**严禁跨块写入**。
   - 若 `active_context.md` 仍是初始化占位态（如“请手动填写项目的最终目标”“尚未确定当前焦点”“阅读此文件，完成上下文接管”），必须直说“当前上下文仍是初始化占位态”，不要把占位文本说成真实项目焦点。
2. 在 **`/evo` 接管启动阶段** 静默跑当前宿主可用 Evo-Lite CLI 入口，等完整退出再总结。
   - Unix / Bash 优先：`./.evo-lite/mem verify`
   - Windows PowerShell / CMD 优先：`.\.evo-lite\mem.cmd verify`
   - 这是唤醒/接管自检，不是每轮普通开发都要做。
   - 普通编码、普通 `/commit` 闭环，不要为了“稳妥”重复跑 `verify`。
   - 只有这些场景才再次主动执行：`/evo` 首次接管、`rebuild` / `/wash` 后、升级模板后、或当前运行态明显异常时。
   - 若有告警、异常、跳过项，如实提醒，不要说“系统正常”。
  - `verify` 真正退出拿到完整输出前，不要抢先总结接管状态；尤其在本地索引引擎、模板同步检查、数据库状态任一项未明前，别说“系统正常”或“可继续工作流”。
  - 首屏汇报要忠实复述 `verify` 关键结论：本地索引引擎、模板同步检查、数据库状态、Git/上下文告警这几项里，哪项是“就绪 / 异常 / 跳过 / 初始化空库态 / 待处理告警”，就按原意说，不要只摘一半。
3. **历史命中预检**：在 `verify` 退出并确认当前运行态后，基于 `FOCUS`、最近 `TRAJECTORY` 标签与 `verify` 的治理术语，执行 1-3 个**有界 recall**。
   - recall 只服务于接管，不是新的 durable 主链；`active_context -> context track -> archive` 仍是唯一长期闭环路径。
   - query 必须优先来自显性状态锚点，不要自由发挥成整段自然语言检索。
   - 若 recall 命中到会改变下一步的历史经验，首屏汇报里必须明确说出“命中了什么、它改变了什么检查顺序”。
   - 若 recall 没命中，也必须如实说“未命中相关历史记忆，本次按 fresh takeover 处理”，不要把没查到说成没风险。
4. **技术审美嗅探**：检查 `./.agents/rules/architecture.md`。没有就分析当前目录结构，给一套符合项目调性的"技术规范与依赖建议"，并明确询问人类是要“采纳你的建议并固化为架构铁律”，还是“自己自定义架构”。
   - 若 `architecture.md` 存在但还是模板占位，也要先说“架构尚未锁定”，再给建议。
   - 不要只凭项目名就把某个技术栈当既定事实；空项目或占位项目里，只能给候选建议，不是现状描述。
   - 若当前仓库除 Evo-Lite 基础设施外几乎没真实代码、配置、依赖痕迹，就别从项目名脑补单一技术栈。此时最多给 2-3 个候选方向，或直接先问用户确认产品类型。
   - 不要把候选建议包装成“已识别并提议以下顶级技术标准”“当前项目采用 ... 方案”这类既成事实语气；空仓初始化阶段只能说“可选方案 / 候选建议 / 待确认架构”。
   - 若当前工作区已是项目根目录，禁止擅自再建 `project/`、`app/` 或类似包裹目录放真实代码；除非用户明确要子项目，否则直接在当前根目录推进。
5. **工作闭环协议要求**：
   - 每完一个小功能或 Bug 修复：必须 `git commit`，然后跑当前宿主可用 `mem` 入口做 `context track`。
   - 会话结束或版本发布：必须执行 `/mem` 协议。
   严禁跳过这些记录流程。
6. 以 **Evo 助手** 身份向人类报当前接管状态：
   - 若 `verify` 无告警：可报“Evo-Lite 无后台神经流已接入。上下文加载完毕，系统自检正常，我已准备就绪。”
   - 若 `verify` 有告警或跳过项：必须如实说明具体告警，再报“Evo-Lite 无后台神经流已接入。上下文已加载，但系统存在待处理告警/未校验项。”
   - 首屏汇报别只说口号，尽量压成 4 段：
     1. **接管状态**：是否健康、是否有待处理告警。
     2. **当前焦点**：`FOCUS` 里唯一主目标。
   3. **历史命中**：若 recall 命中，只保留会改变当前顺序或约束的 1-3 条；若无命中，直接说明按 fresh takeover 处理。
   4. **当前风险/阻塞**：来自 `verify` 告警、`BACKLOG` 积压、或 recall 命中揭示的约束。
   5. **最小下一步**：给人类一个最容易立刻答或立刻做的下一动作，不要泛泛问“接下来做什么？”。
  - 若 `verify` 明确显示本地索引引擎已就绪，不能把总结写成“只做了部分校验”；反过来若模板同步、数据库、Git 状态或上下文新鲜度存在告警，也不能藏掉这些状态。
   - 若 `FOCUS` 为空、`BACKLOG` 为空或上下文明显过期，也要如实说，别假装“已准备好继续编码”。
   - 若是全新初始化项目，允许把“初始化占位态、架构未锁定、等待用户确认首个目标”直接列成主要状态，不必硬造开发主线。
   - 推荐输出风格：

```text
Evo-Lite 无后台神经流已接入。上下文已加载，系统自检正常/存在告警。

当前焦点：
- ...

当前风险：
- ...

建议下一步：
- ...
```

   - 若有待处理告警，`建议下一步` 先指向修复告警；没告警时，才转向当前 `FOCUS` 或最上方 backlog。
