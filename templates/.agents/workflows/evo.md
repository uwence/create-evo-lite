# 🚀 响应 Evo 唤醒协议

当你收到此指令时，代表你已被唤醒执行 Evo-Lite 专属操作。

步骤：
// turbo-all
1. 请立即**静默读取**本项目的 `.evo-lite/active_context.md` 以获悉当前工作进度。
   - 该文件由 HTML 注释锚点划分为四个隔离区块：`META`（项目元数据）、`FOCUS`（当前焦点）、`BACKLOG`（活跃任务 ≤5 条）、`TRAJECTORY`（最近轨迹 ≤10 条）。
   - **锚点守卫**: 后续所有对此文件的写入，必须严格定位到对应锚点区块内操作，**严禁跨区块写入**。
   - 如果你识别到 `active_context.md` 仍处于初始化占位态（例如“请手动填写项目的最终目标”“尚未确定当前焦点”“阅读此文件，完成上下文接管”这类模板文案），必须如实说明“当前上下文仍是初始化占位态”，不要把这些占位文本包装成已经存在的真实项目焦点。
2. 请在 **`/evo` 接管启动阶段** 静默执行当前宿主可用的 Evo-Lite CLI 入口，并等待它完整退出后再总结结果。
   - Unix / Bash 类环境优先使用：`./.evo-lite/mem verify`
   - Windows PowerShell / CMD 环境优先使用：`.\.evo-lite\mem.cmd verify`
   - 这是唤醒/接管时的自检动作，不是每一轮普通开发循环的默认步骤。
   - 普通编码、普通 `/commit` 闭环时，不要为了“看起来稳妥”而重复执行 `verify`。
   - 只有以下场景才应再次主动执行：`/evo` 首次接管、`rebuild` / `/wash` 之后、升级模板之后、或者当前运行态明显异常时。
   - 如果出现告警、异常或跳过项，请如实提醒我，不要把告警说成“系统正常”。
   - 在 `verify` 命令真正退出并拿到完整输出前，不要抢先总结接管状态；尤其不要在 `Embedding / Reranker` 任一状态尚未明确前就宣称“系统正常”或“可继续工作流”。
   - 首屏汇报必须忠实复述 `verify` 的关键结论：`Embedding`、`Reranker`、模板同步检查、数据库状态这四项里，哪一项是“就绪 / 异常 / 跳过 / 初始化空库态”，就按原意说，不要只摘一半，更不要把 `Reranker` 结果省略后改写成“仅确认 Embedding 就绪”。
3. **技术审美嗅探**：请检查本项目的 `./.agents/rules/architecture.md`。如果没有，请分析当前目录结构，给出一套符合本项目调性的"技术规范与依赖建议"，并询问人类是否同意采纳并固定为架构铁律。
   - 如果 `architecture.md` 存在但仍是模板占位内容，也必须先明确说明“架构尚未锁定”，再给建议。
   - 不要仅凭项目名就把某个技术栈说成既定事实；在空项目或占位项目里，你给出的只能是候选建议，不是现状描述。
   - 如果当前仓库除了 Evo-Lite 基础设施外几乎没有真实代码、配置或依赖痕迹，就不要从项目名脑补出单一推荐技术栈。此时最多只能给出 2-3 个候选方向，或者直接请用户先确认产品类型。
   - 不要把候选建议包装成“已识别并提议以下顶级技术标准”“当前项目采用 ... 方案”这类既成事实语气；空仓初始化阶段只能说“可选方案 / 候选建议 / 待确认架构”。
   - 如果当前工作区已经是项目根目录，禁止擅自再创建 `project/`、`app/` 或类似的包裹目录来安放真实代码；除非用户明确要求建立子项目，否则应直接在当前根目录下推进。
4. **工作闭环协议要求**：
   - 每完成一个小功能或 Bug 修复：必须执行 `git commit`，然后运行当前宿主可用的 `mem` 入口来执行 `context track`。
   - 会话结束或版本发布：必须执行 `/mem` 协议。
   严禁跳过上述记录流程。
5. 请以 **Evo 助手** 的身份向人类报告当前接管状态：
   - 若 `verify` 无告警：可以汇报“Evo-Lite 无后台神经流已接入。上下文加载完毕，系统自检正常，我已准备就绪。”
   - 若 `verify` 有告警或跳过项：必须如实说明具体告警，再汇报“Evo-Lite 无后台神经流已接入。上下文已加载，但系统存在待处理告警/未校验项。”
   - 首屏汇报不要只说口号，必须尽量压缩成下面这 4 段：
     1. **接管状态**：是否健康、是否有待处理告警。
     2. **当前焦点**：`FOCUS` 里正在做的唯一主目标。
     3. **当前风险/阻塞**：来自 `verify` 告警、`BACKLOG` 积压或最近轨迹里的风险信号。
     4. **最小下一步**：给人类一个最容易立刻回答或立刻执行的下一动作，而不是泛泛地问“接下来做什么？”。
   - 如果 `verify` 明确显示 `Reranker 引擎状态: 就绪`，你就不能在总结里把系统描述成“只确认 Embedding 引擎就绪”；反过来如果 `Reranker` 异常或跳过，也不能隐去这条状态。
   - 如果 `FOCUS` 为空、`BACKLOG` 为空或上下文明显过期，也必须如实说明，而不是假装“已经准备好继续编码”。
   - 如果当前是全新初始化项目，允许把“初始化占位态、架构未锁定、等待用户确认首个目标”直接列为主要状态，而不是强行伪造一个已经存在的开发主线。
   - 推荐输出风格：

```text
Evo-Lite 无后台神经流已接入。上下文已加载，系统自检正常/存在告警。

当前焦点：
- ...

当前风险：
- ...

建议下一步：
- ...
```

   - 如果存在待处理告警，`建议下一步` 应优先指向修复告警；如果没有告警，才把下一步导向当前 `FOCUS` 或最上方 backlog。
